01

제품 정의 &
포지셔닝

"1 Source → 10 Hooks
Find the highest-converting version in 5 minutes."

HookForge는 어필리에이트 마케터를 위한 영상 A/B 테스트 자동화 SaaS다. 고객은 자기 제품을 홍보하는 원본 영상 하나만 업로드한다. HookForge는 시청자의 시선을 가장 강하게 잡는 첫 3초(Hook)를 열 가지 다른 각도로 자동 생성한 뒤, 원본 본편과 매끄럽게 이어붙여 10개의 완성 영상을 토해낸다.

경쟁사 대비 차별점

Vrew, CapCut은 편집 도구다. 유저가 직접 편집해야 한다.
HeyGen, Opus Clip은 클립 추출/번역 도구다. 같은 영상의 변형이 아니다.
HookForge는 전환율 최적화에 특화된다 — 같은 영상의 첫 3초만 열 가지로 변형해, 어떤 훅이 가장 잘 팔리는지 데이터로 증명한다.

03

영상 레이어
설계도

HookForge의 영상은 2개의 시간 구간과 6개의 레이어로 구성된다. 이 구조가 퀄리티 유지의 핵심이다.

Final Video Structure — 9:16 / 30fps Total ~ 20s
0:00 3s
HOOK
(변동축)
Seedance AI 클립 +
훅 카피 + TTS 더빙
0:03 ~17s
BODY
(고정축)
고객 원본 영상
원본 오디오 유지
Layer 1
Visual Base — Seedance 3초 클립

0~3초 구간의 시각적 배경. Seedance 2.0이 생성한 도파민 클립 (마인크래프트 파쿠르, ASMR, 자동차 묘기 등). 9:16 비율, 1080p, 30fps로 통일.

Layer 2
Hook Audio — ElevenLabs TTS

Claude가 생성한 훅 카피를 ElevenLabs로 더빙. 0~3초 구간에 입혀짐. 보이스 ID 10종 로테이션으로 매번 다른 톤.

Layer 3
Hook Text Overlay — 자막

훅 카피 텍스트를 시각화. 큰 폰트, 노란/흰색 베이스, TikTok 스타일 팝업 애니메이션. 보이스와 정확히 싱크.

Layer 4
Body Visual — 원본 영상

3초 이후 모든 시각 콘텐츠. 고객 원본 영상의 신뢰성과 진정성을 그대로 보존. 9:16으로 자동 크롭/패딩.

Layer 5
Body Audio — 원본 오디오

3초 이후 원본 영상의 오디오 그대로 사용. AI 보이스 없음 — 고객 본인 목소리의 신뢰도 유지.

Layer 6
Transition — 컷 전환

3초 시점에서 매끄러운 컷 전환. 페이드 또는 글리치 효과로 알고리즘 친화적인 갑작스러운 변화 연출. 시청 체류율 상승 트리거.

레이어별 변동/고정 설계

알고리즘 다양성 확보의 핵심은 무엇을 바꾸고 무엇을 유지할 것인가다.

변동축 Layer 1, 2, 3 (Hook 구간 전체) — 10개 버전 100% 다름
고정축 Layer 4, 5 (Body 구간) — 10개 버전 동일
연산 효율 고정 부분 재사용 → 원가 70% 절감
신뢰도 원본 보존 → AI 위화감 최소화
04

퀄리티 유지
정책

자동화에서 가장 흔한 실패는 퀄리티 통제 부재다. HookForge는 모든 렌더링 파라미터를 사전 고정해 결과물 편차를 0에 수렴시킨다.

렌더링 표준 (틱톡 네이티브 최적화)

해상도 1080 × 1920 (9:16 세로형)
프레임 레이트 30fps 강제 통일 (CFR, 가변프레임 금지)
코덱 H.264 (libx264) / AAC 오디오
비트레이트 CRF 23 (틱톡 권장값)
파일 크기 3~5MB / 영상 (15~20초 기준)
오디오 평탄화 LUFS -14 (TikTok 표준)

프레임 정규화 (Frame Normalization)

Seedance 클립(30fps)과 고객 원본 영상(가변, 보통 29.97fps)을 직접 붙이면 프레임 드롭으로 끊김이 발생한다. 병합 직전 두 영상 모두 30fps CFR로 강제 변환한 뒤 합성한다. 이 한 줄이 영상 퀄리티의 80%를 좌우한다.

핵심 파이프라인 코드 구조

from fastapi import FastAPI, BackgroundTasks
import asyncio, httpx
from celery import Celery

app = FastAPI()
celery = Celery(broker="redis://localhost")

@app.post("/generate")
async def create_job(payload: JobInput):
  # 1. 검증 + Job ID 발급
  job_id = create_job_record(payload)
  # 2. 백그라운드 작업 큐에 전송
  run_pipeline.delay(job_id)
  return {"job_id": job_id, "status": "queued"}

@celery.task
async def run_pipeline(job_id):
  # 1. Claude 훅 생성
  hooks = await call_claude(job_id)

  # 2. ElevenLabs + Seedance 완전 병렬
  audios, videos = await asyncio.gather(
    asyncio.gather(*[call_elevenlabs(h) for h in hooks]),
    asyncio.gather(*[call_seedance(h) for h in hooks])
  )

  # 3. Creatomate 합성 병렬
  results = await asyncio.gather(*[
    call_creatomate(audios[i], videos[i], job_id)
    for i in range(10)
  ])

  # 4. 퀄리티 검증 + 이메일
  valid = [r for r in results if validate_video(r)]
  await send_completion_email(job_id, valid)

관계도

users 1:N jobs 1:N hooks 1:1 videos. 한 user는 여러 job을, 한 job은 10개의 hook을, 한 hook은 한 video를 가진다. subscriptions는 user에 1:N (이력 보존용).

인덱싱 전략

jobs (user_id, created_at DESC) — 유저 대시보드 조회
jobs (partial) (status) WHERE status IN ('queued','processing') — 큐 모니터링
videos (job_id) — Job 결과 일괄 조회
subscriptions (stripe_sub_id) UNIQUE — 웹훅 조회 키
users (email) UNIQUE, (stripe_customer_id) — 로그인 / Stripe 조회

Job 상태 머신

Job은 다음 9개 상태 중 하나에 존재한다. 각 전이는 DB에 기록되며 유저 대시보드에 실시간 반영된다. terminal 상태는 3개 — completed / partial / failed.

completed 10/10 영상 검증 통과 — 이메일 발송 + 크레딧 차감
partial 3~9/10 성공 — 성공 영상만 즉시 딜리버리 + 실패 분량 백그라운드 재생성
failed 2/10 이하 또는 fatal 에러 — 크레딧 환불, 최대 3회까지 자동 재시도
10

에러 처리 &
재시도 정책

파이프라인의 6개 단계 각각이 실패할 수 있다. 모든 외부 API 호출은 무조건 재시도 가능하다는 전제로 설계한다. 가장 흔한 실패는 Seedance의 timeout과 ElevenLabs의 rate limit.

API별 재시도 정책

API 실패 시나리오 재시도 폴백
Claude 429, 5xx, timeout 3회 (2s/4s/8s 백오프) 카테고리별 캐시 훅 템플릿
ElevenLabs 429, voice_id 오류 3회 대체 voice_id로 재시도
Seedance (fal) queue full, timeout 180s+ 2회 (60s 간격) Kling 3.0 / Runway Gen-4 라우팅
Creatomate render error, ffmpeg fail 2회 자체 FFmpeg 큐 (Phase 3)
R2 일시적 5xx, network 5회 (1s 간격) 큐 보존, 수동 개입
Stripe webhook 처리 실패 Stripe 자동 재시도 (24h) Slack 알림 + 수동 reconcile

부분 실패 (Partial Failure) 처리

10개 변형 중 일부만 성공하는 케이스가 실무상 가장 흔하다. 임계치별로 명확한 규칙을 둔다.

10/10 성공 → completed — 이메일 발송, 크레딧 차감
8~9/10 성공 → partial — 즉시 딜리버리 + 백그라운드 재생성 (실패 분량)
3~7/10 성공 자동 재생성 1회 후 재평가, 부분 크레딧 차감
0~2/10 성공 → failed — 크레딧 환불, 운영팀 즉시 알림

Idempotency & Dead Letter Queue

모든 Celery 작업은 job_id를 멱등 키로 사용한다. 중복 실행 시 기존 결과를 반환한다. 3회 재시도 후에도 실패한 작업은 DLQ(Dead Letter Queue)로 이동해 운영팀이 수동 검토한다.

@celery.task(bind=True, max_retries=3, default_retry_delay=60)
def call_seedance(self, hook_id):
  try:
    result = fal_client.run("seedance-2.0", prompt=...)
    return result.video_url
  except RateLimitError as e:
    raise self.retry(exc=e, countdown=120)
  except Exception as e:
    if self.request.retries >= 2:
      return fallback_to_kling(hook_id)
    raise self.retry(exc=e)
